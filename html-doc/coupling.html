Loose and tight coupled programs.



MidWay and it's other TP monitor cousins are designed foremost for speed and scaleability, however they do have a place in loosly coupled programs. In order to define what I mean by "loosely coupled" it can be look at degrees of coupling. The most tigthly coupled program are assembly programs with no OS, running straight on the CPU. No memory protection, or defined interfaces.

However consider this rule: The more disjoint the programmers are the smaller and simpler the interfaces must be. Or to make a collorary to Murphy's law, Any API description that may be misunderstood or misused, will. (Ask Microosoft API developers that, especially about the Windows 3.5 to 3.51 upgrade.)

It is not enough to define data types in an interface, since specific values in one may change the legal values in another. Consider printf(3), where even the first argument changes the number of required arguments, not to mention their types. See discussion of OSI layers.

Now I',m gonna say something that is going to be a bit controversial: An object oriented interface is by definition more complex than a functional one.

Consider Corba, in the heart of Corba you have a IDL (Interface Definition Language). You can't even compile you program without having defined it's interfaces. If other objects you use changes their interfaces, you must recompile. The later will normally hold true for MidWay applicatios as well, but you can exchange data encoded like web browsers encode HTML forms data: fieldname1=value&fieldname2=value etc. Then your server may not even know about all the arguents sent it, nor care about them. It allows you to have a much less rigid regime in maintaining versions of all your components.

BTW: I'm not saying that object orientation of your programs is wrong, nor that Corba and it's likes are wrong, quite the contrary, I'm just saying that they are better suited for more tightly coupled systems.
